# RAG 企业知识库项目 -- 分析报告

## 一、项目概述

本项目是一个**RAG（检索增强生成）系统**，最初作为 **[RAG Challenge](https://abdullin.com/ilya/how-to-build-best-rag/) 竞赛的获奖方案**开发，后经改造适配为企业知识库场景，专注于**金融研报/公司年报**的智能问答。系统通过将 PDF 文档解析、向量化、混合检索与 LLM 推理相结合，实现了对中文公司研报的深度理解和精准问答。

> 项目原始仓库：https://github.com/IlyaRice/RAG-Challenge-2  
> 许可证：MIT

### 4.2 数据处理流水线

整个项目的核心流程分为 **离线预处理** 和 **在线推理** 两个阶段：

#### 阶段一：离线预处理（文档索引构建）

```
PDF 研究报告
    │
    ▼
1. PDF 解析 (pdf_parsing.py / pdf_mineru.py)
   使用 Docling 或 MinerU 将 PDF 转为纯 Markdown 文本
   输出: data/stock_data/debug_data/03_reports_markdown/*.md
    │
    ▼
2. 文本分块 (text_splitter.py)
   按行分割 Markdown 文件（默认 chunk_size=30 行，overlap=5 行）
   建立 company_name/sha1 与文件名的映射（基于 subset.csv）
   输出: data/stock_data/databases/chunked_reports/*.json
    │
    ▼
3. 向量化与索引构建 (ingestion.py)
   a) VectorDBIngestor: 调用 DashScope Embedding API 生成向量
      使用 Faiss 构建 IndexFlatIP 向量库（内积 = 余弦距离）
      输出: data/stock_data/databases/vector_dbs/{sha1}.faiss
   b) BM25Ingestor: 构建 BM25 稀疏检索索引
      输出: data/stock_data/databases/bm25_dbs/{sha1}.pkl
```

#### 阶段二：在线推理（问答生成）

```
用户问题
    │
    ▼
1. 公司名提取 (QuestionsProcessor._extract_companies_from_subset)
   从问题文本中匹配 subset.csv 中的公司名，支持多公司
    │
    ▼
2. 检索 (retrieval.py)
   ├── VectorRetriever:   向量检索（默认，基于 Faiss）
   ├── BM25Retriever:     稀疏检索（基于 BM25）
   ├── HybridRetriever:   混合检索 = 向量粗排 + LLM 精排
   │    - 先向量检索 top_k 候选（默认 30）
   │    - 再用 LLM 对候选打分重排
   │    - 加权融合：combined_score = 0.7*LLM_score + 0.3*vector_score
   │
   └── 支持 Parent Document Retrieval：返回完整页面而非分块
    │
    ▼
3. 多公司比较路由 (Query Routing)
   ├── 单公司问题：直接检索 + 生成答案
   └── 多公司比较：先用 LLM 拆分为单公司子问题
       → 并行为每个公司检索+生成 → 再汇总比较生成最终答案
    │
    ▼
4. LLM 推理生成 (api_requests.py + prompts.py)
   ├── 根据问题类型选择 Schema（string/number/boolean/names/comparative）
   ├── Chain-of-Thought 分步推理（step_by_step_analysis）
   ├── 结构化输出（JSON Schema 约束）
   └── 页码验证与引用提取
    │
    ▼
5. 答案后处理
   ├── 页码幻觉检测与过滤（_validate_page_references）
   ├── 1-based → 0-based 页码转换（比赛格式）
   └── 输出 answers.json + answers_debug.json
```

## 五、核心模块详解

### 5.1 Pipeline（流程编排器）

文件：[src/pipeline.py](file:///e:/Python/llm/AI_course/18-项目实战：企业知识库/RAG-cy/src/pipeline.py)

`Pipeline` 类是项目的中枢，负责串联所有处理环节。通过 `PipelineConfig`（路径配置）和 `RunConfig`（运行参数配置）管理不同执行模式。

**RunConfig 关键参数：**

| 参数 | 类型 | 说明 |
|------|------|------|
| `use_serialized_tables` | bool | 是否使用序列化表格 |
| `parent_document_retrieval` | bool | 是否启用父文档检索 |
| `llm_reranking` | bool | 是否启用 LLM 重排 |
| `llm_reranking_sample_size` | int | 初排候选数量（默认 30） |
| `top_n_retrieval` | int | 最终返回文档数（默认 10） |
| `parallel_requests` | int | 并行请求数 |
| `api_provider` | str | LLM 提供商（dashscope/openai/gemini/ibm） |
| `answering_model` | str | 使用的模型名称 |
| `full_context` | bool | 是否使用全文上下文（Gemini 模式） |

**预设配置：**
- `base`: 基础配置，GPT-4o-mini + 向量检索
- `pdr`: 父文档检索 + GPT-4o
- `max`: 父文档检索 + LLM 重排 + Qwen-Turbo（当前默认最佳配置）

### 5.2 检索模块

文件：[src/retrieval.py](file:///e:/Python/llm/AI_course/18-项目实战：企业知识库/RAG-cy/src/retrieval.py)

三种检索器，各具特点：

| 检索器 | 原理 | 适用场景 |
|--------|------|----------|
| `VectorRetriever` | Faiss 向量相似度（余弦距离） | 语义匹配，覆盖同义词 |
| `BM25Retriever` | 词频-逆文档频率稀疏匹配 | 精确关键词匹配 |
| `HybridRetriever` | 向量粗排 + LLM 精排融合 | 追求最高精度 |

### 5.3 提示词工程

文件：[src/prompts.py](file:///e:/Python/llm/AI_course/18-项目实战：企业知识库/RAG-cy/src/prompts.py)

针对不同问题类型设计了专门的 Pydantic Schema：

| Schema | 问题类型 | 答案格式 |
|--------|---------|---------|
| `number` | 数值型 | 精确数字（含单位处理） |
| `name` | 实体名称 | 字符串 |
| `boolean` | 布尔判断 | True/False |
| `names` | 多实体列表 | 字符串数组 |
| `string` | 开放性文本 | 完整段落 |
| `comparative` | 多公司比较 | 公司名称或 N/A |

每个 Schema 都包含 `step_by_step_analysis`（分步推理）、`reasoning_summary`（推理摘要）、`relevant_pages`（引用页码）等结构化字段，确保答案可追溯。

### 5.4 多 API 支持

文件：[src/api_requests.py](file:///e:/Python/llm/AI_course/18-项目实战：企业知识库/RAG-cy/src/api_requests.py)

通过 `APIProcessor` 工厂类统一封装，支持四种 LLM 后端：

| 处理器类 | API 提供商 | 模型示例 |
|----------|-----------|---------|
| `BaseOpenaiProcessor` | OpenAI | GPT-4o, GPT-4o-mini, o3-mini |
| `BaseDashscopeProcessor` | 阿里云 DashScope | Qwen-Turbo |
| `BaseGeminiProcessor` | Google | Gemini 2.0 Flash |
| `BaseIBMAPIProcessor` | IBM Watson | Llama 3.3 70B |

### 5.5 Streamlit 交互界面

文件：[app_streamlit.py](file:///e:/Python/llm/AI_course/18-项目实战：企业知识库/RAG-cy/src/app_streamlit.py)

提供 Web UI 进行单条问题即时推理，展示：
- 分步推理过程（step_by_step_analysis）
- 推理摘要（reasoning_summary）
- 相关页面列表
- 最终答案

## 六、CLI 使用方式

项目通过 `main.py` 提供命令行入口，支持以下命令：

```bash
# 查看帮助
python main.py --help

# 下载 Docling 模型
python main.py download-models

# 解析 PDF（并行模式）
python main.py parse-pdfs --parallel --max-workers 10

# 处理已解析的报告（分块 + 创建向量库）
python main.py process-reports --config no_ser_tab

# 批量处理问题
python main.py process-questions --config max

# 也可直接运行 pipeline.py
python src/pipeline.py
```

## 八、技术亮点

1. **混合检索策略**：向量检索（语义匹配）+ LLM 重排（精准打分），融合两种信号提升检索质量
2. **链式思维推理**：通过 CoT Prompt 引导 LLM 分步推理，提高答案可解释性和准确性
3. **多公司比较路由**：自动识别比较类问题，拆解为单公司子问题后并行处理，再汇总比较
4. **页码幻觉防控**：`_validate_page_references` 方法校验 LLM 引用的页码是否真实存在于检索结果中
5. **多模型适配**：统一接口支持 OpenAI / DashScope / Gemini / IBM 四种 API
6. **并行处理**：多线程问题处理 + 异步 API 调用，提升吞吐量
7. **结构化输出**：使用 Pydantic 定义输出 Schema，确保答案格式规范

## 九、改进建议与注意事项

1. **向量化效率**：当前使用 DashScope Embedding API 单条调用，可考虑批量 Embedding 优化
2. **BM25 检索**：当前默认未启用，可结合向量检索实现 Dense + Sparse 混合检索
3. **流式输出**：Streamlit 前端未实现 SSE 流式推理，需等待完整响应
4. **文档分割策略**：当前按固定行数分块，可考虑按语义边界（段落/章节）分块以提升检索精度
5. **错误处理**：部分模块的异常处理较为粗放，生产环境使用建议加强
6. **依赖管理**：部分依赖（如 docling）需要额外下载模型，首次运行可能较慢
7. **GPU 支持**：PDF 解析阶段使用 Docling 时，GPU 能显著加速（原作者使用 RTX 4090）

## 十、总结

本项目是一个功能完整的 RAG 系统，核心流程为 **PDF 解析 → 文本分块 → 向量化 → 检索 → LLM 生成答案**。项目源自 RAG 竞赛获奖方案，技术实现扎实，涵盖了 RAG 领域的多种进阶技术：

- **检索层面**：向量检索 + BM25 + LLM 重排 + 父文档检索
- **推理层面**：Chain-of-Thought + 结构化输出 + Query Routing
- **工程层面**：多 API 适配 + 并行处理 + CLI/Web 双入口

适合作为企业知识库问答系统的参考实现或基线方案。
